# 纯手动交互式系统测试

本文档用于通过 `run.sh` 启动系统后，在前台 client CLI 中手动测试分布式 MiniSQL 的基本功能。测试过程不使用额外脚本、不设置环境变量、不输入 Python 代码，所有路径、端口和节点名都按当前硬编码配置执行。

## 1. 启动系统

在仓库根目录执行：

```bash
cd /home/ly527609/homework/dis-minisql
./run.sh
```

预期结果：

- MiniSQL 和 Java 分布式层构建成功。
- ZooKeeper 在 `127.0.0.1:12181` 启动。
- `node-a`、`node-b`、`node-c` 依次进入 `SERVING`。
- Coordinator 在后台监听 `127.0.0.1:18080`。
- 终端进入前台 client CLI，并显示 `Connected coordinator: http://127.0.0.1:18080`。

## 2. 查看 CLI 帮助

在 CLI 中输入：

```text
help
```

预期结果：显示 SQL、`nodes`、`metadata`、`health`、`rebalance`、`snapshot`、`quit` 等可用命令。

## 3. 查看节点注册

在 CLI 中输入：

```text
nodes
```

预期结果：能看到 `node-a`、`node-b`、`node-c` 三个节点，状态为 `SERVING`，端口分别是 `19001`、`19002`、`19003`。

如果这里没有节点，先看 `tmp/log/node-a.log`、`tmp/log/node-b.log`、`tmp/log/node-c.log` 是否有端口占用或启动异常。

## 4. 查看分片元数据

在 CLI 中输入：

```text
metadata
```

预期结果：能看到 3 个 shard，每个 shard 有 3 个副本，并且有对应的 `primary`、`term` 和 `commitIndex`。

## 5. 查看集群健康

在 CLI 中输入：

```text
health
```

预期结果：同时返回在线节点和 shard 元数据。这里可以确认节点注册和分片表是否一致。

## 6. 建库建表

在 CLI 中依次输入：

```text
create database dist;
create table t (id int unique, age int, primary key(id));
create table o (id int unique, user_id int, amount int, primary key(id));
```

预期结果：每条命令都返回 `OK: quorum broadcast`。这说明 DDL 已广播到在线 DataNode。

可查看日志：

- `tmp/log/coordinator.log`：应有 Coordinator 启动信息。
- `tmp/log/node-a.log`、`tmp/log/node-b.log`、`tmp/log/node-c.log`：不应出现建表相关异常。

## 7. 测试多数派写入

在 CLI 中依次输入：

```text
insert into t values (1, 10);
insert into t values (2, 20);
insert into t values (3, 30);
insert into o values (101, 1, 7);
insert into o values (102, 2, 9);
```

预期结果：每条写入都返回 `OK: raft write shard ...`，表示写入经过分片路由并达到多数派确认。

可查看每个节点目录中的 WAL：

- `tmp/node-a/wal.jsonl`
- `tmp/node-b/wal.jsonl`
- `tmp/node-c/wal.jsonl`

预期结果：至少能看到写入语句对应的 WAL 记录。

## 8. 测试聚合查询

在 CLI 中输入：

```text
select count(*) from t;
```

预期结果：返回 `OK: scatter-gather read`，查询结果中能看到总数 `3`。

## 9. 测试排序查询

在 CLI 中输入：

```text
select * from t order by age desc;
```

预期结果：返回 `OK: scatter-gather read`，结果按 `age` 从大到小展示，能看到 `30`、`20`、`10`。

## 10. 测试 Join 查询

在 CLI 中输入：

```text
select * from t join o on t.id = o.user_id;
```

预期结果：返回 `OK: coordinator hash join`，结果中能看到订单编号 `101` 和 `102`。

## 11. 测试显式分片键查询

在 CLI 中输入：

```text
shard 1 select * from t;
```

预期结果：返回 `OK: read single shard`，说明 client 将分片键传给 Coordinator，Coordinator 只访问对应分片的一个副本。

## 12. 触发 Snapshot

在 CLI 中输入：

```text
snapshot
```

预期结果：返回当前在线节点列表，表示 Coordinator 已通知在线 DataNode 创建 snapshot。

可查看以下文件是否出现或更新：

- `tmp/node-a/snapshot.sql`
- `tmp/node-b/snapshot.sql`
- `tmp/node-c/snapshot.sql`

## 13. 模拟单节点故障

另开一个终端，在仓库根目录执行：

```bash
kill "$(tr -d '\n' < tmp/pid/node-a.pid)"
```

回到 CLI，输入：

```text
nodes
```

预期结果：等待几秒后，`node-a` 会从在线节点列表中消失，`node-b` 和 `node-c` 仍然在线。

继续在 CLI 中输入：

```text
insert into t values (4, 40);
select count(*) from t;
```

预期结果：写入仍返回 `OK: raft write shard ...`，`count(*)` 能看到总数 `4`。这说明三副本场景下宕掉一个节点后仍可形成多数派。

## 14. 恢复故障节点

在另一个终端执行：

```bash
java -jar dis-minisql/target/dis-minisql-1.0.0.jar datanode ./config/cluster3.json node-a > tmp/log/node-a.log 2>&1 &
echo "$!" > tmp/pid/node-a.pid
```

回到 CLI，等待几秒后输入：

```text
nodes
```

预期结果：`node-a` 重新出现，并进入 `SERVING`。恢复时节点会从其他副本追 WAL。

可查看 `tmp/log/node-a.log`，预期没有恢复失败或端口占用错误。

## 15. 测试扩容和 Rebalance

在另一个终端执行：

```bash
java -jar dis-minisql/target/dis-minisql-1.0.0.jar datanode ./config/cluster4.json node-d > tmp/log/node-d.log 2>&1 &
echo "$!" > tmp/pid/node-d.pid
```

回到 CLI，等待几秒后输入：

```text
nodes
rebalance
metadata
```

预期结果：

- `nodes` 中能看到 `node-d`。
- `rebalance` 返回新的 shard 副本布局。
- `metadata` 中至少有部分 shard 的副本包含 `node-d`。

可查看 `tmp/log/node-d.log`，预期 `node-d` 启动并进入 `SERVING`。

## 16. 扩容后最终查询

在 CLI 中输入：

```text
select count(*) from t;
select * from t order by age desc;
```

预期结果：查询仍成功，`count(*)` 能看到 `4`，排序结果中包含 `40`、`30`、`20`、`10`。

## 17. 查看关键日志

在编辑器中打开以下文件查看：

- `tmp/log/coordinator.log`
- `tmp/log/node-a.log`
- `tmp/log/node-b.log`
- `tmp/log/node-c.log`
- `tmp/log/node-d.log`（如果执行了扩容测试）
- `tmp/log/zk.log`

重点观察：

- Coordinator 是否正常启动在 `127.0.0.1:18080`。
- DataNode 是否进入 `SERVING`。
- 故障恢复时是否有 WAL recovery 相关信息。
- 是否出现 `Address already in use`、`Failed`、`Exception` 等异常。

## 18. 退出 client

在 CLI 中输入：

```text
quit
```

预期结果：只退出前台 client。后台 Coordinator、DataNode 和 ZooKeeper 会继续运行。

## 19. 手动关闭系统

如果不使用任何停止脚本，可以在仓库根目录按固定 PID 文件逐个关闭：

```bash
kill "$(tr -d '\n' < tmp/pid/coordinator.pid)"
kill "$(tr -d '\n' < tmp/pid/node-d.pid)"
kill "$(tr -d '\n' < tmp/pid/node-a.pid)"
kill "$(tr -d '\n' < tmp/pid/node-b.pid)"
kill "$(tr -d '\n' < tmp/pid/node-c.pid)"
kill "$(tr -d '\n' < tmp/pid/zk.pid)"
```

如果没有执行扩容测试，`tmp/pid/node-d.pid` 可能不存在，可以跳过关闭 `node-d` 这一行。

## 20. 测试完成标准

完成以上步骤后，至少应确认：

- `nodes` 能看到在线 DataNode。
- `metadata` 能看到 3 个 shard 和副本信息。
- DDL 广播成功。
- INSERT 返回 `raft write`。
- 聚合、排序、Join 查询成功。
- `snapshot` 能触发在线节点响应。
- 宕掉一个节点后仍能写入和查询。
- 故障节点重启后能重新进入 `SERVING`。
- 新增 `node-d` 后 `rebalance` 能让它进入副本布局。
