# 分布式 MiniSQL 系统测试文档

## 1. 测试目标

本文档说明如何测试整个分布式 MiniSQL 系统，覆盖从基础功能到容错容灾、动态扩容和自动化运维的主要场景。系统由 ZooKeeper、Coordinator 和多个 DataNode 组成，MiniSQL C++ 内核作为 DataNode 的本地 SQL 执行器。

测试重点包括：

- 三节点分片存储和副本写入。
- ZooKeeper 元数据注册和在线节点发现。
- Raft 风格多数派写入。
- 分布式查询、聚合、排序和简单 join。
- 单节点宕机后的读写切换。
- 宕机节点恢复后的 WAL 追赶。
- 新节点加入后的动态扩容和 rebalance。
- snapshot、WAL 压缩和运维接口。

## 3. 自动化端到端测试

推荐优先运行自动化测试脚本：

```bash
python3 dis-minisql/scripts/e2e_system_test.py
```

该脚本会自动完成以下工作：

1. 创建临时运行目录。
2. 使用临时端口启动 ZooKeeper。
3. 启动三个初始 DataNode：`node-a`、`node-b`、`node-c`。
4. 启动 Coordinator。
5. 初始化 ZooKeeper 分片元数据。
6. 执行建库建表。
7. 执行多条 Raft 风格多数派写入。
8. 测试分布式聚合、排序和 join。
9. 停止一个 DataNode，验证剩余多数派仍可继续写入。
10. 启动第四个 DataNode：`node-d`。
11. 调用 `/admin/rebalance`，验证动态扩容。
12. 调用 `/admin/health`，验证运维状态。
13. 调用 `/admin/snapshot`，验证 snapshot 和 WAL 压缩入口。
14. 再次查询，验证故障切换和扩容后数据仍可读。

如果希望保留测试运行目录和日志，执行：

```bash
DIS_MINISQL_KEEP_E2E=1 python3 dis-minisql/scripts/e2e_system_test.py
```

脚本会输出临时目录路径，例如 `/tmp/dis-minisql-e2e-xxxxxx`。其中包含：

- `zk.log`
- `coordinator.log`
- `node-a.log`
- `node-b.log`
- `node-c.log`
- `node-d.log`
- `cluster3.json`
- `cluster4.json`

## 4. 自动化测试覆盖点

### 4.1 基础三节点启动

脚本会启动 ZooKeeper、三个 DataNode 和 Coordinator，并检查各 HTTP 服务是否可访问。

通过条件：

- ZooKeeper 端口可连接。
- 三个 DataNode 的 `/health` 可访问。
- Coordinator 的 `/nodes` 可访问。

### 4.2 建库建表

测试 SQL：

```sql
create database dist;
create table t (id int unique, age int, primary key(id));
create table o (id int unique, user_id int, amount int, primary key(id));
```

通过条件：

- 建库请求广播成功。
- 建表请求广播成功。
- 后续插入和查询可以使用这些表。

### 4.3 Raft 风格多数派写入

测试 SQL：

```sql
insert into t values (1, 10);
insert into t values (2, 20);
insert into t values (3, 30);
insert into o values (101, 1, 7);
insert into o values (102, 2, 9);
```

通过条件：

- Coordinator 返回 `ok = true`。
- 返回的 `route` 中包含 `raft write`。
- 写入需要达到多数派确认后才算成功。

### 4.4 分布式聚合

测试 SQL：

```sql
select count(*) from t;
```

通过条件：

- Coordinator 返回 `mergedOutput`。
- `mergedOutput` 中包含 `3`。

### 4.5 分布式排序

测试 SQL：

```sql
select * from t order by age desc;
```

通过条件：

- Coordinator 返回 `mergedOutput`。
- Coordinator 在合并各分片结果后统一排序。

### 4.6 简化等值 Join

测试 SQL：

```sql
select * from t join o on t.id = o.user_id;
```

通过条件：

- Coordinator 返回 `mergedOutput`。
- `mergedOutput` 中包含订单数据，例如 `101`。
- join 在 Coordinator 内存中完成。

### 4.7 单节点宕机后继续写入

脚本会停止一个 DataNode，然后继续写入：

```sql
insert into t values (4, 40);
```

通过条件：

- 写入仍然成功。
- 因测试配置使用三副本，停止一个节点后仍有两个副本可形成多数派。

### 4.8 动态扩容

脚本会启动第四个节点 `node-d`，然后调用：

```bash
curl -X POST http://127.0.0.1:<coordinator-port>/admin/rebalance
```

通过条件：

- 返回的分片副本列表中包含 `node-d`。
- 新节点加入后会刷新分片并从其他副本追 WAL。

### 4.9 运维接口

健康检查：

```bash
curl http://127.0.0.1:<coordinator-port>/admin/health
```

通过条件：

- 返回在线节点列表。
- 返回所有分片元数据。
- 可看到 primary、term、commitIndex 和副本状态。

触发 snapshot：

```bash
curl -X POST http://127.0.0.1:<coordinator-port>/admin/snapshot
```

通过条件：

- 在线节点收到 snapshot 请求。
- 返回结果数量不少于当前在线节点数。

### 4.10 故障切换和扩容后最终查询

最终测试：

```sql
select count(*) from t;
```

通过条件：

- 返回 `ok = true`。
- `mergedOutput` 中包含 `4`，说明故障期间写入的数据可读。

## 5. 手工测试流程

如果需要手工测试，可以按如下步骤进行。

### 5.1 启动 ZooKeeper

```bash
cd dis-minisql
./scripts/start-zookeeper.sh
```

### 5.2 准备配置

```bash
cp src/main/resources/cluster.example.json cluster.json
```

如果是三台真机部署，需要修改 `cluster.json` 中各节点的 `host` 为对应机器 IP。

### 5.3 初始化元数据

```bash
java -jar target/dis-minisql-1.0.0.jar init-zk cluster.json
```

### 5.4 启动三个 DataNode

分别在三个终端或三台机器上启动：

```bash
./scripts/start-node.sh cluster.json node-a
./scripts/start-node.sh cluster.json node-b
./scripts/start-node.sh cluster.json node-c
```

### 5.5 启动 Coordinator

```bash
./scripts/start-coordinator.sh cluster.json
```

### 5.6 执行基础 SQL

```bash
curl -X POST http://127.0.0.1:8080/sql \
  -H 'Content-Type: application/json' \
  -d '{"sql":"create database dist;"}'

curl -X POST http://127.0.0.1:8080/sql \
  -H 'Content-Type: application/json' \
  -d '{"sql":"create table t (id int unique, age int, primary key(id));"}'

curl -X POST http://127.0.0.1:8080/sql \
  -H 'Content-Type: application/json' \
  -d '{"sql":"insert into t values (1, 10);"}'
```

### 5.7 验证查询

```bash
curl -X POST http://127.0.0.1:8080/sql \
  -H 'Content-Type: application/json' \
  -d '{"sql":"select count(*) from t;"}'
```

检查返回 JSON 中的 `mergedOutput`。

### 5.8 验证节点宕机

停止任意一个 DataNode 进程，然后继续写入：

```bash
curl -X POST http://127.0.0.1:8080/sql \
  -H 'Content-Type: application/json' \
  -d '{"sql":"insert into t values (2, 20);"}'
```

如果副本数为 3，停止 1 个节点后仍然能达到多数派，写入应成功。

### 5.9 验证节点恢复

重新启动刚才停止的 DataNode：

```bash
./scripts/start-node.sh cluster.json node-a
```

恢复节点启动时会进入 `RECOVERING`，从同分片副本拉取 WAL，完成后切换为 `SERVING`。

可通过以下接口观察：

```bash
curl http://127.0.0.1:8080/admin/health
```

### 5.10 验证动态扩容

在配置中添加 `node-d`，启动新节点后调用：

```bash
curl -X POST http://127.0.0.1:8080/admin/rebalance
```

检查返回结果中是否包含 `node-d`。

## 7. 注意事项

- 自动化测试使用临时端口和临时数据目录，不会污染仓库中的 `data/`。
- 如果设置 `DIS_MINISQL_KEEP_E2E=1`，测试临时目录不会删除，便于查看日志。
- 测试中的 Raft 是教学版多数派提交模型，不是完整生产 Raft。
- MiniSQL 内核本身没有事务系统，测试不覆盖“单条 SQL 执行到一半宕机”的事务原子性。
