# 分布式 MiniSQL 系统设计报告

## 1. 目标

本系统在现有 `minisql/` C++ 单机数据库内核外实现一层 Java 分布式系统，满足如下目标：

- 数据按分片键进行哈希分片，分布在多个 DataNode 上。
- 每个分片有多个副本，副本和在线状态由 ZooKeeper 管理。
- Coordinator 提供中心化查询入口，负责分布式路由、读负载均衡和跨分片查询汇总。
- 在 MiniSQL 缺少日志和事务系统的前提下，在 Java 层实现 WAL，用于容错恢复和最终一致副本追赶。
- 支持至少三个真机部署三个数据节点；当一个节点宕机时，Coordinator 自动切换到其他副本。

## 2. 总体架构

系统由三类进程组成：

- Coordinator：中心协调节点，对外暴露 HTTP `/sql` 接口。它读取 ZooKeeper 中的分片元数据和在线节点，负责 SQL 路由。
- DataNode：数据节点，对 Coordinator 暴露 HTTP `/execute`、`/wal`、`/health`。每个 DataNode 维护本地 WAL 和 MiniSQL 数据目录。
- ZooKeeper：保存集群元数据和在线节点。DataNode 使用 ephemeral znode 注册，节点宕机后会自动从在线列表消失。

MiniSQL 内核保持不变，由 Java DataNode 通过子进程方式调用。这样可以把分布式逻辑、WAL、容错恢复全部放在 `minisql/` 外部，符合“数据库内核无日志和事务系统，无需实现分布式事务”的要求。

## 3. 元数据设计

ZooKeeper 路径如下：

- `/dis-minisql/nodes/<nodeId>`：DataNode 在线信息，ephemeral 节点，内容为 `NodeInfo` JSON。
- `/dis-minisql/shards/<shardId>`：分片副本信息，persistent 节点，内容为 `ShardMetadata` JSON。
- `/dis-minisql/shard-log-indexes/<shardId>`：每个分片的递增日志序号，由 Coordinator 在写入前分配。

分片初始化采用确定性轮转方式。例如三节点、三分片、副本数为二时：

- shard 0: node-a, node-b
- shard 1: node-b, node-c
- shard 2: node-c, node-a

该方式保证副本均匀分布，且无需额外主节点选举即可进行简化部署。

## 4. 分片与复制

Coordinator 选择分片键后计算：

```text
shardId = sha256(shardKey) mod shardCount
```

对于 `insert into ... values (...)`，如果用户未显式传入 `shardKey`，系统默认取第一列值作为分片键；也可以在 HTTP 请求中传入 `shardKey`。

写请求流程：

1. Coordinator 根据分片键定位 shard。
2. 从 ZooKeeper 获取该 shard 的副本列表和当前在线节点。
3. 为该 shard 分配递增 `shardLogIndex`。
4. 检查该 shard 的 primary 和 term；如果 primary 不在线，则从在线 `SERVING` 副本中选出新 primary 并递增 term。
5. 将写请求发送给所有在线且状态为 `SERVING` 的副本。
6. DataNode 先追加本地 `wal.jsonl`，再调用 MiniSQL batch 模式执行。
7. 只有成功副本数达到多数派时，Coordinator 才推进 shard 的 `commitIndex` 并返回成功；失败副本恢复后通过 WAL 追赶。

DDL 语句如 `create table`、`drop table` 会广播到所有在线 DataNode，保证各分片本地 schema 一致。

## 5. WAL 与容错恢复

DataNode 的 WAL 为 JSON Lines 文件，每条记录包含：

- `sequence`：本节点递增序列号。
- `requestId`：Coordinator 为一次写入生成的全局请求 ID。同一次写入发往多个副本时使用相同 ID，用于恢复去重。
- `shardId`：该条日志所属分片，DDL 广播日志使用 `-1`。
- `shardLogIndex`：该分片上的递增日志序号，用于观察副本追赶进度。
- `sql`：规范化后的 SQL。
- `timestampMillis`：写入时间。

DataNode 维护 `snapshot.json` 和 `snapshots/current/snapshot.sql`。snapshot 记录已经反映到逻辑状态中的最大 WAL 序号和各分片最大 `shardLogIndex`。由于 MiniSQL 内核的行数据跨进程恢复能力有限，Java 分布式层把 WAL/snapshot SQL 作为权威逻辑状态；每次 batch 执行在临时工作目录中用 snapshot SQL 和增量 WAL 重建状态，再执行本次 SQL。WAL 增长到一定数量后，DataNode 会生成新的 snapshot SQL，并压缩已经包含进 snapshot 的 WAL。

恢复流程：

1. DataNode 启动后以 `RECOVERING` 状态在 ZooKeeper 注册。
2. 根据 ZooKeeper 分片表找到与自己有共同分片的在线副本。
3. 调用对方 `/wal` 接口拉取 WAL。
4. 只保留 `shardId` 属于本节点负责分片或 `shardId = -1` 的广播日志。
5. 按 `requestId` 去重后追加本地缺失日志。
6. 恢复本地 snapshot，并应用 snapshot 之后的 WAL 增量。
7. 恢复完成后切换为 `SERVING`，Coordinator 才会选择它处理读写请求。

该设计面向“节点无正在执行任务时宕机，恢复后补执行离线期间其他副本已执行操作”的场景。它不处理单条 SQL 执行到一半时进程崩溃的事务原子性问题。系统实现最终一致性；强一致性需要多数派提交、日志复制协议或事务提交协议，考虑到原 MiniSQL 无事务系统，本实现采用更适合作业规模的 GFS/BigTable 风格简化方案。

## 6. 查询执行

读请求支持两种模式：

- 指定 `shardKey`：Coordinator 只访问对应分片的一个在线副本。
- 不指定 `shardKey`：Coordinator 对所有分片做 scatter-gather，每个分片选择一个在线副本执行，最终把多个节点的输出一起返回。

负载均衡使用 per-shard 轮询策略。每次读同一分片时，Coordinator 会在在线副本之间轮转，避免读请求集中到单个节点。

复杂查询采用中心化协调模式。Coordinator 会解析 MiniSQL 的表格输出，把各分片 `select` 结果合并为一个逻辑结果表；对于 `order by`，Coordinator 会去掉下发 SQL 中的排序部分，在汇总后统一排序；对于 `count/sum/min/max`，Coordinator 会改写为各分片 `select *` 并在中心节点计算聚合值；对于形如 `select * from A join B on A.x = B.y` 的简单等值 join，Coordinator 会分别拉取两张表的分片结果并执行内存 hash join。该实现仍是简化查询引擎，不覆盖完整 SQL 优化器能力。

## 7. 部署

Java 工程位于 `dis-minisql/`，示例配置为 `dis-minisql/src/main/resources/cluster.example.json`。

构建 MiniSQL：

```bash
cd minisql
mkdir -p build
cd build
cmake ..
make -j
```

构建 Java：

```bash
cd dis-minisql
mvn package
```

启动：

```bash
./scripts/start-zookeeper.sh
java -jar target/dis-minisql-1.0.0.jar init-zk cluster.json
./scripts/start-node.sh cluster.json node-a
./scripts/start-node.sh cluster.json node-b
./scripts/start-node.sh cluster.json node-c
./scripts/start-coordinator.sh cluster.json
```

三台真机部署时，只需要把 `cluster.json` 中每个节点的 `host` 改为对应机器 IP，并在每台机器启动自己的 DataNode。

## 8. 动态扩容与自动化运维

Coordinator 提供 `/admin/health`、`/admin/rebalance` 和 `/admin/snapshot`：

- `/admin/health` 返回当前在线节点、分片副本、primary、term、commitIndex 和各副本进度。
- `/admin/rebalance` 根据当前在线节点重新计算每个分片的副本集合；新 DataNode 加入后，调用该接口即可把部分分片迁移到新节点。Coordinator 随后通知各 DataNode 刷新分片，DataNode 进入 `RECOVERING`，从同分片副本拉取 WAL 并追赶，完成后切回 `SERVING`。
- `/admin/snapshot` 通知所有在线 DataNode 生成 snapshot SQL，并压缩 WAL。

## 9. 系统测试

项目提供 `dis-minisql/scripts/e2e_system_test.py`。该脚本自动启动临时 ZooKeeper、三个初始 DataNode 和 Coordinator，并覆盖：

- 建库建表和 MiniSQL batch 执行。
- Raft 风格多数派写入。
- `count/sum/min/max`、`order by` 和简化等值 join。
- 单节点宕机后继续写入。
- 第四个 DataNode 加入后的动态 rebalance。
- `/admin/health` 和 `/admin/snapshot` 运维接口。

## 10. 局限与改进

- 当前 Raft 是教学版多数派提交模型，包含 primary、term、commitIndex 和多数派确认，但不包含完整 Raft 的选举超时、日志回滚和成员变更联合共识。
- WAL/snapshot 以 SQL 重放为核心，适合教学 MiniSQL；生产系统应使用更严格的幂等日志、LSN、物理快照和校验。
- 当前复杂查询支持基础聚合、排序和简单等值 join，尚未实现完整 SQL 优化器。
- 动态扩容支持在线 rebalance 和副本追赶，但迁移策略仍是简单轮转分布，尚未支持热点感知和一致性哈希。
