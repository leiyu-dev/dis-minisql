# 分布式 MiniSQL — 项目设计总览

本文档从**整体架构、核心机制、模块边界、元数据与通信**等维度概括本仓库的设计，便于快速建立全局认识。更细的总体方案见 [`总体设计报告.md`](./总体设计报告.md)，分模块类级说明见 [`模块设计报告.md`](./模块设计报告.md)，编译与启动见根目录 [`README.md`](../README.md)。

---

## 1. 项目定位与系统边界

### 1.1 定位

在 **MiniSQL（C++ 单机数据库内核）** 之上，用 **Java** 实现一层分布式中间件：**协调节点（Coordinator）** 负责 SQL 解析与路由、元数据与集群视图；**数据节点（DataNode）** 每个进程内嵌一个 **minisql 子进程**，负责实际存储与执行；**ZooKeeper** 持久化集群与分片元数据，并用临时节点做成员发现；**客户端** 通过 TCP 与 Coordinator 交互。

### 1.2 系统边界（做什么 / 不做什么）

| 能力 | 说明 |
|------|------|
| 数据分布 | 按**分片键**（由建表 DDL 解析出的主键列）做哈希取模，映射到固定数量的分片；分片数与建表时存活节点数相关。 |
| 元数据 | 库、表结构、分片主副本分配等写入 ZK；`SHOW DATABASES` / `SHOW TABLES` 可走 ZK，不访问 DataNode。 |
| 副本与写入 | 写主节点成功后，Coordinator **异步**向副本节点下发 `REPLICATE`（尽力而为，非强一致事务）。 |
| 容错 | 节点会话失效后，Coordinator 在分片元数据中**晋升可用副本**为新主（见 `ClusterManager`）。 |
| 复杂查询 | 支持在 Coordinator 侧对两表做**内存哈希 JOIN**（全量拉取后连接）。 |
| 明确不做 | 跨节点 **ACID 事务**、在线 resharding、Coordinator 自身 HA 等（见总体设计报告「权衡与局限」）。 |

## 2. 逻辑架构

```
                    ┌─────────────┐
                    │   Client    │  交互式 CLI，会话级 currentDb
                    └──────┬──────┘
                           │ TCP：长度前缀 JSON（SqlRequest / SqlResponse）
                    ┌──────▼──────────────────────────────────────┐
                    │            Coordinator                       │
                    │  ClientHandler（会话）→ SqlRouter（路由）     │
                    │  MetadataManager（表/分片 + ZK）            │
                    │  ClusterManager（成员 + 故障转移）           │
                    │  ResultMerger（多路结果 / JOIN）               │
                    └──────┬────────────┬────────────┬──────────────┘
                           │            │            │  短连接 EXECUTE / REPLICATE
              ┌────────────▼─┐  ┌───────▼──────┐  ┌──▼────────────┐
              │  DataNode    │  │  DataNode    │  │  DataNode     │
              │ SqlExecutor  │  │ SqlExecutor  │  │ SqlExecutor   │
              │ MinisqlProcess│ │ MinisqlProcess│ │ MinisqlProcess │
              └──────┬───────┘  └───────┬──────┘  └──┬────────────┘
                     │ stdin/stdout     │                  │
              ┌──────▼───────┐  ┌───────▼──────┐  ┌──▼────────────┐
              │   minisql    │  │   minisql    │  │   minisql     │  C++ 单机引擎
              └──────────────┘  └──────────────┘  └───────────────┘
                           所有 Java 进程通过 Curator 访问
                    ┌──────▼──────────────────────────────────────┐
                    │              ZooKeeper                        │
                    │  /dis-minisql/{nodes,databases,tables,shards}│
                    └─────────────────────────────────────────────┘
```

**要点**：对客户端而言只有一个「逻辑数据库」入口；物理上数据按分片落在多个 minisql 实例上，由 Coordinator 根据 SQL 类型决定广播、单分片或扇出。

## 3. 核心设计机制

### 3.1 分片与分片键

- **分片数 `numShards`**：建表时取当前**存活节点数**（至少为 1），写入 `TableMeta`。
- **分片键**：从 `CREATE TABLE` 中解析 **PRIMARY KEY**（列级或表级约束）；若无主键声明，则退化为第一列（见 `MetadataManager`）。
- **分片 ID**：`ShardingStrategy.getShardId(keyValue, numShards)` 对键的字符串形式取 `hashCode()` 再 `abs % numShards`（与数值主键的重载并存）。

### 3.2 分片到节点的映射（主 + 副本）

建表成功后，`MetadataManager` 为每个 `shardId` 构造 `ShardInfo`：

- **主节点**：`liveNodeIds.get(shardId % n)`（`n` 为节点数）。
- **副本列表**：在环上顺延 `replicationFactor - 1` 个不同节点（见 `buildReplicaList`）。

副本集合与 `Coordinator` / `DataNode` 启动参数 `--replication-factor` 一致才有意义。

### 3.3 SQL 路由策略（SqlRouter）

与实现注释一致，概括为：

| 类别 | 行为 |
|------|------|
| `SHOW DATABASES` | 读 ZK `databases`，不访问 DataNode。 |
| `SHOW TABLES` | 读 ZK `tables/{db}`，需当前库。 |
| `CREATE/DROP DATABASE` | 广播到所有存活节点；成功后更新 ZK 库集合。 |
| `USE db` | 广播到所有节点（保证各 minisql 切换当前库）；会话在 Coordinator 维护。 |
| DDL（表/索引等） | 一般**广播**，`CREATE/DROP TABLE` 成功后再写/删 ZK 中表元与分片信息。 |
| `INSERT` | 解析主键值 → 算分片 → 发主节点；成功后 `replicateAsync`。 |
| `SELECT` / `UPDATE` / `DELETE` | WHERE 中存在**分片键等值**时走**单分片**；否则 **fan-out** 到全节点并合并（`ResultMerger`）。 |
| 带 `JOIN` 的 `SELECT` | 由路由层识别后拉取参与表的全量数据，在协调节点内存做哈希连接。 |
| 无法解析的 SQL | 记录告警后**广播**（降级为各节点自行执行）。 |

**读负载**：单分片只读路径上通过 `pickReadNode` 在 **primary + replicas** 中选一个存活节点；实现上以 `System.nanoTime()` 在候选环上滑动起点，等价于随机/轮询式负载分散（非严格全局计数器）。

### 3.4 副本写入与 REPLICATE

- 客户端写操作先到**分片主**对应 DataNode。
- Coordinator 在写成功后向 `ShardInfo` 中的副本 ID 列表异步提交 `MessageType.REPLICATE` 请求；对端 `ReplicaManager` / `ReplicaClient` 处理，保证主从之间**最终一致**语义（无同步复制屏障）。

### 3.5 故障检测与转移（ClusterManager）

- DataNode 在 ZK 上注册 **`/dis-minisql/nodes/{nodeId}` 临时节点**；会话超时或进程退出导致 znode 消失。
- `ClusterManager.updateNodes` 对比前后成员集，对「离开」的节点调用 `handleNodeFailure`：扫描 ZK 中所有库表分片，将 `primaryNodeId` 仍为失效节点的分片**改写为第一个仍存活的副本**（并更新 ZK）。

### 3.6 与 MiniSQL 进程的集成（DataNode）

- `MinisqlProcess` 用 `ProcessBuilder` 启动子进程，工作目录为**该节点独立**的 `data-dir/nodeId`，避免文件冲突。
- 协议：向 stdin 写入 `SQL;\n`，通过 stdout 上出现 **`minisql > `** 提示符判断一条命令结束（见 `MinisqlProcess.waitForPromptFrom`）。
- `SqlExecutor` 在 `database` 上下文变化时先发 `use <db>;`（小写以匹配 C 端词法）；`MinisqlOutputParser` 将多种文本输出归类为成功/失败并抽取表格或影响行数，供封装为 `SqlResponse`。

**与单机层的契约**：发往 minisql 的语句需符合其词法（例如关键字小写、字符串双引号等）；分布式层用 JSQLParser 做路由侧解析，与 C 端解析器相互独立。

## 4. ZooKeeper 数据模型（与代码一致）

路径常量定义于 `com.minisql.common.zk.ZkPaths`：

| 路径 | 类型 | 含义 |
|------|------|------|
| `/dis-minisql/nodes/{nodeId}` | ephemeral | `NodeInfo` JSON，成员与探活。 |
| `/dis-minisql/coordinator` | （常量已定义） | `ZkPaths.COORDINATOR` 预留供协调者注册/HA 感知；当前 `CoordinatorZkManager.start()` 仅确保 `ROOT` 与子目录并监听 `nodes`，未强制写入该 znode。 |
| `/dis-minisql/databases/{dbName}` | persistent | 库存在标记。 |
| `/dis-minisql/tables/{dbName}/{tableName}` | persistent | `TableMeta` JSON（分片键、列、建表 SQL、分片数等）。 |
| `/dis-minisql/shards/{dbName}/{tableName}/{shardId}` | persistent | `ShardInfo` JSON（主节点、副本列表）。 |

DataNode 启动时会 `creatingParentsIfNeeded` 创建根路径下的目录结构；**多节点并发首次创建根** 在极端竞态下可能触发 `NodeExists`，实践上可错开启动顺序或先由单节点创建。

## 5. 通信与协议（common）

- **帧格式**：4 字节大端长度 + UTF-8 JSON（`MessageCodec`）。
- **SqlRequest**：`requestId`、`type`（`EXECUTE` / `REPLICATE` / `PING`）、`sql`、`database`。
- **SqlResponse**：`status`、`message`、`rawOutput`、`columns`、`rows`、`affectedRows` 等，供 CLI 表格渲染。

Client 与 Coordinator 为**长连接**；Coordinator 到 DataNode 多为**短连接**按请求建立（实现以 `Socket` + 线程池扇出为主）。

## 6. Maven 模块职责

| 模块 | 职责 |
|------|------|
| **common** | 协议模型、ZK 路径、JSON 工具、minisql 文本输出解析、网络编解码。 |
| **coordinator** | 接入客户端、`SqlRouter` 路由与合并、`MetadataManager` / `ClusterManager`、Curator 访问 ZK。 |
| **datanode** | 监听请求、`SqlExecutor` / `MinisqlProcess`、ZK 注册与节点监听、`ReplicaManager` 处理复制。 |
| **client** | 交互式 CLI，`MiniSQLClient` 维护 `currentDb` 并格式化 `SqlResponse`。 |
| **minisql** | 第三方提供的 C++ 内核；分布式层通过子进程调用，不在本仓库的 Java 改造范围内修改业务语义。 |

## 7. 关键技术栈（摘要）

- Java 11、Maven 多模块。
- **Apache Curator**：ZK 会话、路径缓存与监听。
- **Jackson**：节点与元数据 JSON 序列化。
- **JSQLParser**：Coordinator 侧 SQL 分类、主键/WHERE 提取、JOIN 检测。
- **Logback**：日志。

## 8. 设计权衡与阅读建议

- **一致性**：异步复制 + 无分布式事务 → 低延迟、实现简单，极端情况下副本可能短暂滞后或丢失末次复制（需在课程/报告中说明）。
- **扩展性**：JOIN 与无主键筛选类查询依赖**全分片数据**在协调节点聚合，适合教学与小数据验证；生产级需 shuffle、下推谓词等优化。
- **单 Coordinator**：逻辑单点；高可用需在 ZK 上选主或多活路由层扩展。

**建议阅读顺序**：本总览 → [`总体设计报告.md`](./总体设计报告.md)（数据流与时序）→ [`模块设计报告.md`](./模块设计报告.md)（类与调用关系）→ 结合 `coordinator/.../SqlRouter.java`、`datanode/.../SqlExecutor.java` 对照源码。
