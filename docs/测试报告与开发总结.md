## 二、测试用例

### 2.1 基本功能测试

#### TC-01：数据库创建与切换

```sql
create database testdb;
show databases;
use testdb;
```

**预期**：`SHOW DATABASES` 返回包含 `testdb` 的结果行；`USE testdb` 返回 `Query OK`。

---

#### TC-02：建表与元数据注册

```sql
use testdb;
create table users (
    id   int,
    name varchar(50),
    age  int,
    primary key(id)
);
show tables;
```

**预期**：
- `CREATE TABLE` 广播到 3 个节点，每个节点均建表成功。
- ZooKeeper `/dis-minisql/tables/testdb/users` 节点被创建，包含 `shardKey=id, numShards=3`。
- `/dis-minisql/shards/testdb/users/{0,1,2}` 3 个分片节点被创建。
- `SHOW TABLES` 返回 `users`。

---

#### TC-03：分片写入（INSERT 路由验证）

```sql
-- shardId = |hash("1")| % 3
insert into users values (1, 'Alice', 20);
insert into users values (2, 'Bob',   25);
insert into users values (3, 'Carol', 22);
insert into users values (4, 'Dave',  30);
insert into users values (5, 'Eve',   28);
```

**验证方式**：直接向各 DataNode（端口 9001/9002/9003）发送  
`SELECT * FROM users;` 检查各节点实际存储的行。  
每个节点存储的行号应满足 `|hash(id)| % 3 == nodeIndex`。

---

#### TC-04：主键等值 SELECT（单分片路由）

```sql
select * from users where id = 1;
select * from users where id = 4;
```

**预期**：每次只路由到一个 DataNode，返回正确的单行结果。

---

#### TC-05：全表扫描（扇出合并）

```sql
select * from users;
```

**预期**：结果集包含全部 5 行，来自 3 个节点的部分结果被正确合并。

---

#### TC-06：条件过滤（非主键，扇出）

```sql
select * from users where age > 24;
```

**预期**：广播到所有节点，各节点执行过滤，协调节点合并，返回 age > 24 的所有行。

---

#### TC-07：UPDATE / DELETE 路由

```sql
-- 主键等值条件 → 单分片
update users set age = 21 where id = 1;

-- 无主键条件 → 扇出
delete from users where age > 29;
```

**预期**：
- 前者只修改一个节点上 id=1 的行，受影响行数 = 1。
- 后者在所有节点执行 DELETE，累加受影响行数。

---

#### TC-08：索引创建

```sql
create index idx_age on users(age);
```

**预期**：DDL 广播，3 个节点均创建索引成功。

---

### 2.2 JOIN 测试

#### TC-09：INNER JOIN

```sql
create table orders (
    oid     int,
    user_id int,
    amount  float,
    primary key(oid)
);

insert into orders values (1001, 1, 99.9);
insert into orders values (1002, 2, 150.0);
insert into orders values (1003, 1, 50.0);
insert into orders values (1004, 3, 200.0);

select users.name, orders.amount
from users join orders on users.id = orders.user_id;
```

**预期**：返回 users 与 orders 的等值连接结果（4 行），列为 `[name, amount]`（加上两表全列）。

---

### 2.3 副本与容错测试

#### TC-10：副本同步验证

向 node1 写入若干行后，直连 node2（副本）查询同一表，确认数据已同步（延迟 < 1s）。

---

#### TC-11：主节点故障转移

```
步骤：
1. 记录 shard-0 的 primary = node1（ZK 确认）
2. 强制关闭 node1 进程
3. 等待 ~15 秒（ZK session timeout）
4. 查询 /dis-minisql/shards/testdb/users/0，确认 primaryNodeId 已更新为 node2
5. 执行 INSERT INTO users VALUES (10, 'Grace', 35);
6. 验证写入成功（路由到新 primary node2）
```

**预期**：node1 宕机后，Coordinator 的 ClusterManager 自动检测故障，将 shard-0 的 primary 更新为 node2。后续写入/读取均可正常进行。

---

#### TC-12：负载均衡验证

连续执行 30 次 `SELECT * FROM users WHERE id = 1`，记录每次实际到达的 DataNode（通过日志）。  
**预期**：请求在 primary(node1) 和 replica(node2) 之间轮询分发。

---

### 2.4 并发测试

#### TC-13：并发写入

使用 5 个客户端线程，每线程写入 100 条不同 id 的记录。  
**预期**：总写入量 = 500，最终 `SELECT COUNT(*) FROM users`（或全表 SELECT 行数）= 原始行数 + 500，无数据丢失、无重复。

---

#### TC-14：并发读写

5 个写入线程 + 5 个查询线程同时运行。  
**预期**：读取不出现异常，写入全部成功，最终数据一致。

## 五、开发总结

### 5.1 项目亮点

1. **分层解耦**：C++ 内核（存储引擎）与 Java 分布式层严格分离，通过 stdin/stdout 管道通信，互不干扰。

2. **ZooKeeper 全量元数据管理**：所有集群状态（节点注册、表结构、分片分配）均持久化于 ZooKeeper，Coordinator 重启后无需重新感知数据分布，具备良好的容错基础。

3. **自动故障转移**：利用 ZooKeeper Ephemeral ZNode + CuratorCache Watch，实现节点宕机的秒级感知与副本自动晋升，无需人工干预。

4. **协调节点内存 JOIN**：通过哈希 JOIN 算法在协调节点完成跨分片连接查询，支持基本的分布式复杂查询需求。

5. **负载均衡读**：读请求在主节点和副本节点之间轮询，充分利用副本资源，提升读吞吐量。

### 5.2 遇到的挑战

1. **minisql 子进程输出解析**  
   minisql 以文本方式输出结果，需要准确判断每条命令的输出边界（以 `minisql > ` 提示符为分隔）。实现了基于 `wait()/notifyAll()` 的异步读取机制，避免了忙等待和死锁问题。

2. **SQL 路由中的主键提取**  
   JSQLParser 对 INSERT 语句的 VALUES 解析需要区分「位置参数」与「命名列」两种情况，并正确找到分片键在 VALUES 列表中的位置。对 WHERE 子句需要递归分析 AND 树，提取等值条件。

3. **分布式 JOIN 的内存限制**  
   当前 JOIN 实现将两张表的全量数据拉到协调节点内存。对于大表 JOIN 场景，这可能成为瓶颈。未来可以改进为：基于分片键的 Partition-wise JOIN（仅拉取匹配分片数据），或流式归并 JOIN。

4. **异步副本与一致性**  
   采用异步副本复制意味着在主节点宕机的极短窗口内（最后一批写入尚未同步到副本），可能有少量数据丢失。对于本课程的实验场景，这是可接受的权衡；生产环境中可改用同步复制或 Raft 日志。

### 5.3 后续优化方向

| 方向                   | 说明                                                      |
|------------------------|-----------------------------------------------------------|
| 协调节点高可用          | 引入 ZooKeeper Leader Election，支持多 Coordinator 主备切换 |
| 在线 Resharding         | 增加节点时重新分配分片，支持动态扩容                       |
| 查询优化                | 下推 WHERE 过滤（仅送入相关列），减少网络传输量            |
| 分布式 JOIN 优化        | Partition-wise JOIN、Broadcast Hash Join（小表广播）      |
| 数据恢复                | 节点重启后从副本拉取缺失数据，实现数据修复                 |
| 监控与运维              | 暴露 JMX / HTTP 健康接口，集成 Prometheus + Grafana        |
