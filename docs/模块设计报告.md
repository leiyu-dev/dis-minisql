## 一、公共模块（common）

### 1.1 模块职责

`common` 模块是所有其他 Java 模块的依赖基础，提供：
- TCP 消息编解码（`MessageCodec`）
- 请求/响应协议对象（`SqlRequest`、`SqlResponse`）
- ZooKeeper 路径常量（`ZkPaths`）
- 领域模型（`NodeInfo`、`TableMeta`、`ShardInfo`）
- minisql 输出解析工具（`MinisqlOutputParser`）
- JSON 序列化工具（`JsonUtil`）

### 1.2 类图

```
protocol/
├── MessageType            枚举: EXECUTE, REPLICATE, PING, PONG
├── SqlRequest             请求: requestId, type, sql, database
└── SqlResponse            响应: requestId, status, rawOutput, columns, rows, affectedRows

model/
├── NodeInfo               节点信息: nodeId, host, port, lastHeartbeat, status
├── TableMeta              表元数据: dbName, tableName, shardKey, shardKeyIndex, numShards, columns, createSql
└── ShardInfo              分片信息: shardId, dbName, tableName, primaryNodeId, replicaNodeIds

zk/
└── ZkPaths                ZK 路径常量与构建方法

util/
├── JsonUtil               Jackson 封装: toJson, fromJson, toBytes, fromBytes
└── MinisqlOutputParser    解析 minisql 文本输出为 ParsedResult

net/
└── MessageCodec           长度前缀 JSON 帧编解码器
```

### 1.3 关键设计：MinisqlOutputParser

minisql 的输出分三类，`MinisqlOutputParser.parse(String raw)` 统一处理：

```
输入 raw 字符串
        │
        ├─ 含 "+"  且含 "|" → 解析表格输出 → ParsedResult(columns, rows)
        ├─ 含 "Query OK"    → 解析受影响行数 → ParsedResult(affectedRows)
        ├─ 含 "Empty set"   → 返回空结果集
        └─ 其他             → 判定为错误，errorMessage = raw
```

表格解析逻辑：按行扫描，跳过 `+---+` 分隔符行，第一个 `|...|` 行为列名，其余为数据行，以 `|` 拆分单元格后 `trim()`。

### 1.4 关键设计：MessageCodec

```
写入:  [4字节大端序长度] + [UTF-8 JSON 字节]
读取:  readInt() → 分配 byte[] → readFully() → UTF-8 解码 → Jackson 反序列化
```

使用 `DataOutputStream / DataInputStream` + `BufferedStream` 实现，每次 `flush()` 确保数据发送。

## 二、数据节点模块（datanode）

### 2.1 模块职责

DataNode 是 MiniSQL 底层存储能力的网络封装。它：
1. 启动并管理一个 minisql 子进程。
2. 监听来自 Coordinator 的 TCP 连接，每个连接由一个线程处理。
3. 将收到的 SQL 请求转给 minisql 执行，返回原始输出。
4. 向 ZooKeeper 注册自身为活跃节点（ephemeral ZNode）。
5. 监听集群变化，维护副本客户端列表。

### 2.2 类图

```
DataNodeMain                     程序入口，组装依赖，启动服务
DataNodeConfig                   命令行参数解析与校验
DataNodeServer                   ServerSocket 主循环，接受连接 → 派发 RequestHandler
DataNodeZkManager                ZK 注册 / 节点监听 / 反通知 ReplicaManager
handler/
└── RequestHandler               处理单个连接：读 SqlRequest → 调 SqlExecutor → 写 SqlResponse
executor/
├── MinisqlProcess               minisql 子进程生命周期管理 + stdin/stdout 交互
└── SqlExecutor                  数据库切换 + 委托 MinisqlProcess 执行 + 解析输出
replica/
├── ReplicaManager               维护 nodeId→ReplicaClient 映射，提供 replicateWrite / broadcastWrite
└── ReplicaClient                连接单个远端节点，同步或异步发送 REPLICATE 请求
```

### 2.3 MinisqlProcess 时序图

```
DataNodeServer     MinisqlProcess     minisql(C++)
      │                  │                  │
      │   start()        │                  │
      │─────────────────▶│  ProcessBuilder  │
      │                  │──────────────────▶ 启动子进程
      │                  │  readLoop 线程   │
      │                  │◀─ "minisql > " ──│  (初始提示符)
      │                  │  waitForPrompt() │
      │                  │  waitForPrompt 返回
      │                  │                  │
      │  execute(sql)    │                  │
      │─────────────────▶│                  │
      │                  │──write "sql;\n"──▶
      │                  │                  │  (执行SQL)
      │                  │◀─ 输出 + "minisql > " ─│
      │                  │  waitForPrompt 返回 rawOutput
      │◀─ rawOutput ─────│                  │
```

关键实现细节：
- 使用 `StringBuilder buffer + Object lock` 实现阻塞读取。
- `readLoop` 线程每次读到数据后调用 `lock.notifyAll()`。
- `waitForPromptFrom(startPos)` 在 `synchronized(lock)` 内循环检查 `buffer.indexOf("minisql > ", startPos)`，找到则返回，否则 `lock.wait()`。
- `execute()` 方法以 `synchronized(this)` 确保单线程串行（minisql 本身不支持并发）。

### 2.4 副本复制时序图

```
Coordinator       DataNode(primary)      DataNode(replica)
    │                    │                      │
    │  EXECUTE INSERT    │                      │
    │───────────────────▶│                      │
    │                    │ execute via minisql  │
    │                    │──────────────────────┤
    │                    │ SUCCESS              │
    │◀───────────────────│                      │
    │                    │                      │
    │                    │ REPLICATE (async)    │
    │                    │─────────────────────▶│
    │                    │                      │ execute via minisql
```

## 三、协调节点模块（coordinator）

### 3.1 模块职责

Coordinator 是整个系统的"大脑"，负责：
1. 接受客户端 SQL 连接，维护会话状态（当前数据库）。
2. 使用 JSQLParser 解析 SQL，确定路由策略。
3. 向 DataNode 发送请求（单播或广播），并行或串行。
4. 合并多节点响应（行集合并、受影响行数累加、哈希 JOIN）。
5. 通过 ZooKeeper 维护集群视图，处理节点故障转移。
6. 在 ZooKeeper 中持久化表结构和分片分配元数据。

### 3.2 类图

```
CoordinatorMain                  程序入口
CoordinatorConfig                参数解析
CoordinatorServer                ServerSocket 主循环
handler/
└── ClientHandler                会话线程：维护 currentDb，调 SqlRouter.route()
meta/
├── ClusterManager               liveNodes 维护，故障检测，触发副本晋升
└── MetadataManager              建表时提取主键/分片键，写 ZK；查表/分片元数据
router/
├── SqlRouter                    SQL 路由引擎（核心，见下）
└── ResultMerger                 行集合并、受影响行数聚合、哈希 JOIN 实现
shard/
└── ShardingStrategy             getShardId(keyValue, numShards)
zk/
└── CoordinatorZkManager         ZK CRUD 封装：节点/数据库/表/分片的读写与监听
```

### 3.3 SqlRouter 路由决策流程图

```
route(requestId, database, sql)
        │
        ├─ SHOW DATABASES ──────────────────────▶ 直接查 ZK 返回
        ├─ SHOW TABLES ─────────────────────────▶ 直接查 ZK 返回
        ├─ CREATE DATABASE ─────────────────────▶ 广播 + ZK 注册
        ├─ DROP DATABASE ───────────────────────▶ 广播 + ZK 删除
        ├─ USE database ────────────────────────▶ 广播（更新所有节点上下文）
        │
        │   [JSQLParser.parse()]
        │
        ├─ CreateTable ─────────────────────────▶ 广播 + ZK 保存 TableMeta + ShardInfo
        ├─ Drop(TABLE) ─────────────────────────▶ 广播 + ZK 删除元数据
        ├─ CreateIndex ─────────────────────────▶ 广播
        │
        ├─ Insert ──▶ 提取主键值
        │             │
        │             ├─ 成功 ──▶ shardId = hash(pk) % numShards
        │             │           ──▶ 路由到对应 primary
        │             │           ──▶ 异步复制到 replicas
        │             └─ 失败 ──▶ 发送到 node0（兜底）
        │
        ├─ Select
        │   ├─ 含 JOIN ──▶ handleJoin()：拉两表全量数据，内存哈希 JOIN
        │   ├─ WHERE 含主键等值 ──▶ 单分片路由（读负载均衡：主/副本轮询）
        │   └─ 其他 ──▶ 扇出到全部节点，mergeRows()
        │
        ├─ Update ──▶ WHERE 含主键等值 ──▶ 单分片写 + 副本复制
        │             否则 ──▶ 全节点扇出，mergeAffected()
        │
        └─ Delete ──▶ 同 Update
```

### 3.4 哈希 JOIN 流程图

```
handleJoin(requestId, database, sql, plainSelect)
    │
    ├─ 解析左表名 (leftTable)、右表名 (rightTable)
    ├─ 解析 ON A.col = B.col 获取 leftJoinCol / rightJoinCol
    │
    ├─ fanOutAndMergeRows("SELECT * FROM leftTable")  → leftResp
    ├─ fanOutAndMergeRows("SELECT * FROM rightTable") → rightResp
    │
    └─ ResultMerger.hashJoin(
           leftCols, leftRows,
           rightCols, rightRows,
           leftJoinCol, rightJoinCol)
           │
           ├─ 在 rightRows 上按 rightJoinCol 建 HashMap<key, List<row>>
           ├─ 遍历 leftRows，对每行取 leftJoinCol 值
           ├─ 在 HashMap 中查找匹配的 rightRows
           └─ 拼接 leftRow + rightRow → 输出行集合
```

### 3.5 故障转移时序图

```
DataNode-1    ZooKeeper     CoordinatorZkManager   ClusterManager
    │              │                 │                    │
    │  (宕机)      │                 │                    │
    │  Session 超时│                 │                    │
    │──(断连)─────▶│                 │                    │
    │              │ delete ephemeral│                    │
    │              │ /nodes/node1    │                    │
    │              │─────────────────▶ watch callback     │
    │              │                 │──updateNodes(live)─▶
    │              │                 │                    │ 发现 node1 消失
    │              │                 │                    │ handleNodeFailure("node1")
    │              │                 │                    │ 找到 shard(primary=node1)
    │              │                 │◀──promoteToPrimary─│
    │              │  更新 /shards/  │                    │
    │              │◀───────────────▶│                    │
    │              │                 │                    │
    │              │                 │  后续路由跳过 node1，使用新 primary
```

## 四、客户端模块（client）

### 4.1 模块职责

提供交互式命令行界面（CLI），让用户输入 SQL 语句并查看结果。

### 4.2 类图

```
ClientMain          程序入口，参数解析，交互主循环
MiniSQLClient       TCP 连接封装：维护 currentDb，发送 SqlRequest，接收 SqlResponse
                    提供 format(SqlResponse) 静态方法，渲染表格输出
```

### 4.3 交互时序

```
用户              ClientMain            MiniSQLClient        Coordinator
 │                    │                      │                    │
 │  输入 SQL;          │                      │                    │
 │───────────────────▶│                      │                    │
 │                    │  execute(sql)        │                    │
 │                    │─────────────────────▶│                    │
 │                    │                      │  writeRequest()    │
 │                    │                      │───────────────────▶│
 │                    │                      │                    │ 路由/执行
 │                    │                      │◀── readResponse() ─│
 │                    │◀─ SqlResponse ───────│                    │
 │                    │  format(resp)        │                    │
 │◀─ 格式化表格 ───────│                      │                    │
```

### 4.4 多行 SQL 支持

客户端采用"累积直到遇到 `;`"的策略：

```
读取一行
    │
    ├─ 含 ';' ──▶ 追加到 buffer，发送 buffer 内容，清空 buffer
    └─ 不含 ';' ──▶ 追加到 buffer，继续读取下一行（显示续行提示符 "->"）
```

## 五、关键数据流：INSERT 端到端

以 `INSERT INTO users VALUES (42, 'Alice')` 为例：

```
Client
  └─▶ Coordinator (ClientHandler)
        ├─ currentDb = "testdb"
        └─▶ SqlRouter.route("testdb", "INSERT INTO users VALUES (42, 'Alice')")
              ├─ JSQLParser.parse() → Insert 对象
              ├─ getTableMeta("testdb","users") → meta (shardKey="id", numShards=3)
              ├─ extractInsertKeyValue(insert, meta) → "42"
              ├─ ShardingStrategy.getShardId("42", 3) → 42 % 3 = 0
              ├─ getAllShards().find(shardId=0) → ShardInfo(primary="node1",replicas=["node2"])
              ├─ sendToNode("INSERT ...", "node1") ─────────────────────────────┐
              │                                                                  ▼
              │                                                     DataNode-1 (node1)
              │                                                       RequestHandler
              │                                                       SqlExecutor.execute("testdb","INSERT...")
              │                                                       MinisqlProcess.execute("use testdb; INSERT...")
              │                                                       ← "Query OK, 1 row affected"
              │                                                       ← SqlResponse(SUCCESS, affectedRows=1)
              ├─◀ SUCCESS
              ├─ replicateAsync("testdb","INSERT...", ["node2"]) ────────────────┐
              │                                                                   ▼
              │                                                     DataNode-2 (node2, async)
              └─▶ 返回 SUCCESS 给 Client
```

## 六、SELECT ... JOIN 端到端

以 `SELECT u.id, o.amount FROM users u JOIN orders o ON u.id = o.user_id` 为例（3 节点）：

```
SqlRouter.handleJoin()
  ├─ 扇出 "SELECT * FROM users"  ─────▶ [node1, node2, node3] 并行
  │     结果: columns=[id,name], rows=[(1,Alice),(4,Dave),...]  ← 合并
  ├─ 扇出 "SELECT * FROM orders" ─────▶ [node1, node2, node3] 并行
  │     结果: columns=[id,user_id,amount], rows=[...]           ← 合并
  └─ ResultMerger.hashJoin(
         usersRows, ordersRows,
         joinCol_left="id", joinCol_right="user_id")
         │
         ├─ Build: HashMap{ user_id → [orderRows] } 来自 orders
         ├─ Probe: 遍历 users，每行取 id，查 HashMap
         └─ 输出合并列: [id, name, order_id, user_id, amount]
```

## 七、技术选型说明

### ZooKeeper + Curator

选用 Apache Curator（ZooKeeper 高层客户端）的原因：
- 内置连接管理、重连重试（`ExponentialBackoffRetry`）。
- `CuratorCache`（替代旧版 PathChildrenCache）提供响应式节点监听。
- 提供 Leader Election、分布式锁等 Recipe，便于后续扩展协调节点高可用。

### JSQLParser

选用 JSQLParser 进行 SQL 解析的原因：
- 纯 Java 实现，无需重新实现词法/语法分析。
- 支持 INSERT/SELECT/UPDATE/DELETE/CREATE TABLE/DROP 等完整 SQL 语法。
- 提供对象化 AST（`Insert`、`Select`、`PlainSelect`、`Join` 等），便于提取列名、WHERE 条件、JOIN 关系。

### 消息协议：长度前缀 JSON

- JSON 可读性好，便于调试。
- 长度前缀帧解决 TCP 粘包/拆包问题（不依赖特殊分隔符）。
- Jackson 在 Java 生态中性能和兼容性俱佳。
- 若未来需要性能优化，可替换为 Protobuf/Avro。
