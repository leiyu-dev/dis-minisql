## 一、系统背景与目标

MiniSQL 是一个使用 C++ 实现的单机关系型数据库内核，支持基本的 DDL（CREATE/DROP DATABASE/TABLE/INDEX）和 DML（INSERT/SELECT/UPDATE/DELETE）操作，并提供 B+ 树索引、缓冲池管理等存储引擎功能。

本项目的目标是以 MiniSQL 为底层存储内核，在其上构建一套具备以下能力的分布式数据库系统：

| 目标特性         | 说明                                                     |
|------------------|----------------------------------------------------------|
| **数据分布**     | 以哈希分片方式将表的行数据横向切分到多个数据节点         |
| **集群管理**     | 通过 ZooKeeper 完成节点注册、发现与健康监控              |
| **分布式查询**   | 协调节点解析 SQL，路由到正确分片并聚合结果               |
| **副本管理**     | 每个分片可配置多个副本，保证数据持久性                   |
| **容错容灾**     | 节点宕机时自动将副本晋升为主节点，屏蔽对客户端的影响     |
| **负载均衡**     | 读请求在主副本间轮询分发                                 |
| **复杂查询**     | 协调节点内存哈希 JOIN，支持跨分片连接查询                |

## 三、数据分布设计

### 3.1 分片策略

采用 **哈希分片（Hash Partitioning）**，以表主键列为分片键。

```
shardId = |hash(primaryKeyValue)| % numShards
```

- `numShards` = 建表时集群中活跃的 DataNode 数量（建表后固定）。  
- 每个 DataNode 拥有一个 minisql 进程，存储其对应分片的行数据。  
- 所有节点的表结构（Schema）完全相同（DDL 广播）；数据按主键哈希分布。

### 3.2 分片元数据（ZooKeeper）

```
/dis-minisql/shards/{dbName}/{tableName}/{shardId}
```

内容（JSON）：

```json
{
  "shardId":       0,
  "dbName":        "testdb",
  "tableName":     "users",
  "primaryNodeId": "node1",
  "replicaNodeIds":["node2"]
}
```

### 3.3 路由规则

| SQL 类型                      | 路由策略                                       |
|-------------------------------|------------------------------------------------|
| DDL（CREATE/DROP TABLE 等）   | **广播**到所有 DataNode                        |
| INSERT                        | 提取主键值 → 计算 shardId → 路由到对应主节点  |
| SELECT（主键等值条件）         | 同 INSERT 路由逻辑 → 单节点读                  |
| SELECT（无主键条件）           | **扇出**到所有节点 → 合并去重返回             |
| SELECT ... JOIN               | 拉取两表全量数据 → 协调节点内存哈希 JOIN      |
| UPDATE / DELETE（主键等值）    | 单节点写 → 同步副本                           |
| UPDATE / DELETE（无主键条件）  | 扇出到所有节点 → 累加受影响行数               |

## 四、集群管理设计

### 4.1 节点注册与发现

每个 DataNode 在 ZooKeeper 上创建一个 **Ephemeral ZNode**：

```
/dis-minisql/nodes/{nodeId}
```

ZNode 的值为 JSON 序列化的 `NodeInfo`（包含 host、port、心跳时间）。  
Coordinator 通过 `CuratorCache` 监听该目录，当节点加入或离开时触发回调。

```
时序：DataNode 启动
  DataNode  ──create ephemeral──▶  ZooKeeper
  ZooKeeper ──watch event──────▶  Coordinator
  Coordinator ──更新 ClusterManager.liveNodes──▶ 路由表生效
```

### 4.2 健康检测

- DataNode 到 ZooKeeper 的 **Session 超时**（默认 15 秒）用于故障检测。  
- ZooKeeper Session 断开时 Ephemeral ZNode 自动删除，Coordinator 立即感知。  
- 可选：Coordinator 也可定期对 DataNode 发送 PING 请求做主动探活。

### 4.3 容错与故障转移

```
故障检测时序：
  DataNode-1 宕机
  ZooKeeper 删除 /dis-minisql/nodes/node1 (session timeout)
  Coordinator.ClusterManager 收到 watch 事件
  ClusterManager 遍历所有分片，找到 primaryNodeId=node1 的分片
  对每个此类分片：
    选出第一个 alive 的副本节点（如 node2）
    调用 ZK 更新 /shards/.../{shardId} 中的 primaryNodeId = node2
  后续写请求路由到 node2
```

## 五、副本管理设计

### 5.1 副本分配

建表时，设分片总数为 N，节点顺序为 [n0, n1, n2, ...]，对于 shardId=i：

```
primary  = nodes[i % N]
replica1 = nodes[(i+1) % N]
replica2 = nodes[(i+2) % N]   (若 replicationFactor >= 3)
```

### 5.2 写复制协议

采用 **主节点同步写 + 异步副本复制** 策略：

```
Client ──INSERT──▶ Coordinator
Coordinator ──路由到 primary──▶ DataNode-primary ──写 minisql──▶ 成功
DataNode-primary ──响应──▶ Coordinator ──响应──▶ Client
Coordinator ──异步 REPLICATE──▶ DataNode-replica（最终一致）
```

- 主节点写入成功即向客户端返回。  
- 副本复制为异步、尽力而为（best-effort）。  
- 副本延迟不影响写入延迟，但在极端情况下可能丢失少量数据。  
- 无事务系统要求，不需要 2PC/Paxos 等分布式事务协议。

### 5.3 读负载均衡

读请求（SELECT）在可用节点（primary + replicas）中**轮询**分发，以均摊读取压力。

## 六、分布式查询设计

### 6.1 查询执行流程

```
Client → Coordinator
         │
         ├─ 解析 SQL 类型
         ├─ 查 ZK 确定目标分片
         ├─ 并行向 DataNode 发送请求
         ├─ 等待全部响应（或超时）
         └─ 合并结果 → 返回 Client
```

### 6.2 JOIN 查询实现

对于 `SELECT A.* , B.* FROM A JOIN B ON A.id = B.a_id`：

1. Coordinator 检测到 JOIN 子句。  
2. 分别向所有节点扇出 `SELECT * FROM A` 和 `SELECT * FROM B`，收集全部数据。  
3. 在协调节点内存中以较小的表为探针端（Build），建立哈希表。  
4. 以较大的表为驱动端（Probe），遍历查找匹配行。  
5. 拼接左右列，返回合并结果集。

```
哈希 JOIN 示意（内存中）：

  左表 A 的全部行 ─────────────────┐
                                    ▼
  右表 B 的全部行 ──▶ HashMap(B.a_id → [行]) ──▶ 对 A 的每行探针 ──▶ 输出
```

> **局限性**：当前 JOIN 仅支持等值连接（INNER JOIN），且全量数据需装入协调节点内存。大表 JOIN 可在未来优化为分区 JOIN 或 MapReduce 式分布式 JOIN。

## 七、通信协议

### 7.1 消息格式

所有节点间通信使用 **长度前缀 JSON 帧**（Length-Prefixed JSON Frame）：

```
┌────────────────────────────────────────┐
│  4 字节 (int, big-endian): payload 长度 │
├────────────────────────────────────────┤
│  N 字节 UTF-8 JSON                     │
└────────────────────────────────────────┘
```

### 7.2 请求/响应结构

**SqlRequest**（Client→Coordinator 或 Coordinator→DataNode）：

```json
{
  "requestId": "uuid",
  "type":      "EXECUTE | REPLICATE | PING",
  "sql":       "SELECT * FROM users WHERE id=1;",
  "database":  "testdb"
}
```

**SqlResponse**（DataNode→Coordinator 或 Coordinator→Client）：

```json
{
  "requestId":    "uuid",
  "status":       "SUCCESS | ERROR",
  "message":      "...",
  "rawOutput":    "+----+------+\n| id | name |\n...",
  "columns":      ["id", "name"],
  "rows":         [["1", "Alice"], ["2", "Bob"]],
  "affectedRows": 2
}
```

### 7.3 DataNode 与 minisql 子进程通信

DataNode（Java）通过 **stdin/stdout** 与 minisql（C++）进程交互：

```
Java DataNode ──写 "SQL;\n" ──▶ minisql stdin
minisql ──执行 SQL──▶ 输出结果到 stdout
minisql ──打印提示符 "minisql > "──▶ Java 读取，以此检测命令完成
```

`MinisqlProcess` 使用后台线程持续读取 stdout，通过 `wait()/notifyAll()` 在检测到提示符后唤醒等待的执行线程。

## 十、系统设计权衡与局限

| 方面           | 当前设计                          | 未来改进方向                      |
|----------------|-----------------------------------|-----------------------------------|
| 副本一致性     | 异步复制（最终一致）               | Semi-sync 或 Raft 日志复制        |
| 分片键固定     | 建表时确定，不可变更               | 在线 resharding                   |
| JOIN 扩展性    | 全量数据拉到协调节点内存合并       | Partition-wise JOIN / shuffle     |
| 协调节点 HA    | 单 Coordinator，单点风险           | 主备 Coordinator + ZK 选主        |
| 事务           | 不支持（MiniSQL 无事务系统）       | 两阶段锁 / MVCC                   |
| 分片算法       | 哈希分片（固定 bucket 数）         | 一致性哈希 / Range 分片            |
