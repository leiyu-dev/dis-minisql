# 分布式 MiniSQL：从启动到各模块功能测试全流程指南

本文说明从**环境准备 → 编译 → 启动 ZooKeeper 与集群 → 按模块验证功能**的完整操作顺序，并与仓库内 [`测试报告与开发总结.md`](./测试报告与开发总结.md) 中的用例编号对应。SQL 示例与 **MiniSQL C 端词法**一致：**关键字小写**、字符串用**双引号**、字符类型用 **`char(n)`**（无 `varchar`）。

---

## 一、前置条件

| 项目 | 说明 |
|------|------|
| 操作系统 | Linux / macOS（与 MiniSQL 构建说明一致） |
| JDK | 11 |
| Maven | 3.8+ |
| ZooKeeper | 3.8+ 单机或集群；客户端连接串默认 `localhost:2181` |
| 端口 | `2181`（ZK）、`8080`（Coordinator，可改）、`9001–9003`（示例 DataNode） |

---

## 二、获取代码与可选清理

```bash
cd /path/to/dis-minisql
```

若需**全新环境**（避免旧元数据干扰）：

```bash
# 清空 ZK 上本项目命名空间（ZooKeeper 须已启动且可连）
printf 'deleteall /dis-minisql\nquit\n' | zkCli.sh -server localhost:2181

# 删除各节点本地数据目录（与 start-datanode.sh 中 data-dir 下 nodeId 一致）
rm -rf data/node1 data/node2 data/node3
```

---

## 三、编译 MiniSQL（C++ 内核）

```bash
cd minisql
mkdir -p build && cd build
cmake .. && make -j4
cd ../..
```

**可执行文件位置**：当前 CMake 生成的主程序为 **`minisql/build/bin/main`**，而 `start-datanode.sh` 默认查找 **`minisql/build/minisql`**。请在项目根目录执行一次：

```bash
ln -sf bin/main minisql/build/minisql
```

若 `ln` 后 `test -f minisql/build/minisql` 为真，即可被启动脚本识别。

---

## 四、编译 Java（含 common / datanode / coordinator / client）

在项目根目录：

```bash
mvn clean package -DskipTests
```

修改过 `common` 等被 fat-jar 打入的模块后，务必 **`clean package`**，避免运行到旧的 class。

成功后可检查：

- `datanode/target/datanode-1.0.0-jar-with-dependencies.jar`
- `coordinator/target/coordinator-1.0.0-jar-with-dependencies.jar`
- `client/target/client-1.0.0-jar-with-dependencies.jar`

---

## 五、配置并启动 ZooKeeper

1. 在 ZK 安装目录的 **`conf/zoo.cfg`** 中配置 **`dataDir`**（勿仅用 `/tmp` 做长期数据）、**`clientPort=2181`**。  
2. 启动：

```bash
/path/to/apache-zookeeper-*-bin/bin/zkServer.sh start
/path/to/apache-zookeeper-*-bin/bin/zkServer.sh status
```

若使用本仓库旁解压的 ZK，可将 `zkServer.sh` 换为实际路径。

---

## 六、启动 DataNode（datanode 模块）

**脚本必须在项目根目录下执行**（或始终使用绝对路径调用 `start-datanode.sh`），否则找不到 JAR / minisql。

**建议错开几秒启动多个节点**，避免多进程同时 `create` ZK 根路径时的竞态（见设计文档说明）：

```bash
ROOT=/path/to/dis-minisql   # 改为你的仓库根路径

$ROOT/start-datanode.sh node1 9001 &
sleep 5
$ROOT/start-datanode.sh node2 9002 &
sleep 5
$ROOT/start-datanode.sh node3 9003 &
```

默认 **`--replication-factor=2`**（与 `start-datanode.sh` 内写死值一致），与三节点、`start-coordinator.sh` 的第三个参数为 `2` 时一致。

**验证 DataNode**：日志中出现 `Connected to ZooKeeper`、`Registered DataNode`、`listening on port`；或：

```bash
printf 'ls /dis-minisql/nodes\nquit\n' | zkCli.sh -server localhost:2181
```

应能看到 `node1`、`node2`、`node3` 子节点（会话有效时）。

---

## 七、启动 Coordinator（coordinator 模块）

```bash
$ROOT/start-coordinator.sh 8080 localhost:2181 2
# 参数：Coordinator 监听端口  ZK 连接串  replication-factor
```

若 `8080` 已被占用，可改为 `18080` 等，**客户端连接端口需一致**。

**验证 Coordinator**：日志中出现 `Coordinator listening`、`Node joined`；客户端能 `ping` 通（见下一节）。

---

## 八、启动 Client（client 模块）与连接性

交互式：

```bash
$ROOT/start-client.sh localhost 8080
```

启动后应先看到连接成功说明；模块级最小验证：

- 输入 **`\status`**：确认当前连接与 `Current database`（无需发 SQL）。  
- 输入 **`show databases;`**：走 Coordinator + ZK，不经过 DataNode 读库表数据。

退出：`\q` 或 `quit`。

---

## 九、按功能用例完成「各模块」相关测试

下列与 **TC-01～TC-09** 等对应；在 **client** 中逐条执行（每条以 **`;`** 结束）。**模块覆盖说明**写在每节标题后。

### 9.1 TC-01：库级 DDL + 会话（Coordinator 元数据 + 广播 + Client 会话）

**覆盖**：`MetadataManager` / ZK `databases`；`USE` 广播到各 DataNode；`MiniSQLClient` 维护 `currentDb`。

```sql
show databases;
create database testdb;
use testdb;
```

**预期**：`show databases` 出现 `testdb`；`use testdb` 成功（或显示等价成功信息）。

**可选 ZK 核对**：

```bash
printf 'ls /dis-minisql/databases\nquit\n' | zkCli.sh -server localhost:2181
```

---

### 9.2 TC-02：建表与 ZK 元数据（Coordinator + DataNode + ZK）

**覆盖**：DDL 广播、`MetadataManager.registerTable`、分片写入 ZK。

```sql
create table users (
    id   int,
    name char(32),
    age  int,
    primary key(id)
);
show tables;
```

**预期**：`show tables` 含 `users`。

**ZK 核对**（路径与 `ZkPaths` 一致）：

```bash
printf 'ls /dis-minisql/tables/testdb\nls /dis-minisql/shards/testdb/users\nquit\n' | zkCli.sh -server localhost:2181
```

应存在 `users` 及 `0`、`1`、`2`… 等分片节点（分片数 = 建表时存活节点数）。

---

### 9.3 TC-03：分片写入（SqlRouter + ShardingStrategy + DataNode）

**覆盖**：`INSERT` 按主键哈希路由到主节点、异步副本（日志层可观察）。

```sql
insert into users values (1, "Alice", 20);
insert into users values (2, "Bob",   25);
insert into users values (3, "Carol", 22);
insert into users values (4, "Dave",  30);
insert into users values (5, "Eve",   28);
```

**说明**：字符串须为 **双引号**，以符合 MiniSQL 词法。

**深度验证（可选）**：对 `9001/9002/9003` 用独立 TCP 客户端或自写脚本向各 DataNode 发 `select * from users;`（协议同 `SqlRequest`），观察各行是否仅出现在哈希对应节点（教学/报告用）。

---

### 9.4 TC-04～TC-05：主键点查与全表扫描（路由 + ResultMerger）

```sql
select * from users where id = 1;
select * from users;
```

**预期**：点查返回单行；全表返回全部已插入行。**注意**：全表扇出合并时，若多副本上均存有相同逻辑行，结果行数可能多于逻辑行数，以课程/实现说明为准。

---

### 9.5 TC-06：非主键条件扇出

```sql
select * from users where age > 24;
```

**预期**：返回 age 大于 24 的所有逻辑行（合并后）。

---

### 9.6 TC-07：UPDATE / DELETE 路由

```sql
update users set age = 21 where id = 1;
delete from users where age > 29;
```

**预期**：主键等值 `update` 影响一行；无分片键等值条件的 `delete` 扇出，受影响行数为各节点之和（可能大于逻辑删除行数，与实现一致）。

---

### 9.7 TC-08：索引 DDL（广播）

```sql
create index idx_age on users(age);
```

**预期**：无报错；各节点 catalog 上存在索引（可通过各节点执行 `show indexes` 类语句验证，若 MiniSQL 版本支持）。

---

### 9.8 TC-09：JOIN（Coordinator 内存哈希 JOIN）

```sql
create table orders (
    oid     int,
    user_id int,
    amount  float,
    primary key(oid)
);

insert into orders values (1001, 1, 99.9);
insert into orders values (1002, 2, 150.0);
insert into orders values (1003, 1, 50.0);
insert into orders values (1004, 3, 200.0);

select users.name, orders.amount
from users join orders on users.id = orders.user_id;
```

**预期**：得到等值连接结果集（行数与数据一致）。

---

### 9.9 TC-10～TC-12：副本、故障转移、读均衡（进阶）

与 [`测试报告与开发总结.md`](./测试报告与开发总结.md) **2.3 节**一致：

- **TC-10**：写入后主从延迟内，在副本节点上可读同一主键数据。  
- **TC-11**：杀掉某一 DataNode 进程，等待 ZK session 超时后，检查 `shards` 下对应分片 `primaryNodeId` 是否被 `ClusterManager` 更新，再执行 `insert` 验证新主。  
- **TC-12**：对同一主键等值 `select` 多次，结合 Coordinator/DataNode 日志观察读是否分散到多副本（实现为基于时间的候选轮换，非严格计数器）。

---

### 9.10 TC-13～TC-14：并发（可选）

使用多进程/多线程各自启动 `client` jar 或自写并发发送 `SqlRequest`，按测试报告中的线程数与断言验证。

## 十、common 模块如何算「测到」

`common` 无独立进程，其 **`MessageCodec`、`SqlRequest`/`SqlResponse`、`MinisqlOutputParser`** 随 **DataNode** 与 **Client↔Coordinator** 全链路被调用。

**最小证明**：Client 能持续收发且 `show databases` / `insert` 返回结构化表格或 `Query OK`，即协议与解析链路工作正常。若单独调试解析，可在本机对 `MinisqlOutputParser` 写单元测试（非本仓库必选）。

## 十一、推荐执行顺序小结

| 步骤 | 动作 |
|------|------|
| 1 | 安装并启动 ZK，确认 `2181` 可连 |
| 2 | 编译 `minisql` 并建立 `minisql/build/minisql` → `bin/main` 的符号链接 |
| 3 | `mvn clean package -DskipTests` |
| 4 | 错开启动 3 个 `start-datanode.sh` |
| 5 | `start-coordinator.sh` |
| 6 | `start-client.sh`，按 §9 顺序执行 SQL |
| 7 | （可选）`zkCli` 核对 ZK；进阶执行 TC-10～TC-14 |

## 十二、常见问题

| 现象 | 处理方向 |
|------|----------|
| 找不到 `minisql/build/minisql` | 按 §三 建立指向 `bin/main` 的符号链接 |
| DataNode 报 ZK `NodeExists` | 多个节点同时冷启动竞态；停掉后错开重启，或先 `deleteall /dis-minisql` |
| `USE` 后仍报库切换失败 | 确认已 `mvn clean package` 使用最新 `common`（`MinisqlOutputParser` 需识别 `Database changed` 等成功输出） |
| MiniSQL 语法错误 | 关键字小写、字符串双引号、类型用 `int`/`float`/`char(n)` |
| Coordinator 端口占用 | 换端口并同步修改 `start-client.sh` 参数 |
