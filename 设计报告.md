# 分布式 MiniSQL 系统设计报告

## 1. 目标

本系统在现有 `minisql/` C++ 单机数据库内核外实现一层 Java 分布式系统，满足如下目标：

- 数据按分片键进行哈希分片，分布在多个 DataNode 上。
- 每个分片有多个副本，副本和在线状态由 ZooKeeper 管理。
- Coordinator 提供中心化查询入口，负责分布式路由、读负载均衡和跨分片查询汇总。
- 在 MiniSQL 缺少日志和事务系统的前提下，在 Java 层实现 WAL，用于容错恢复和最终一致副本追赶。
- 支持至少三个真机部署三个数据节点；当一个节点宕机时，Coordinator 自动切换到其他副本。

## 2. 总体架构

系统由三类进程组成：

- Coordinator：中心协调节点，对外暴露 HTTP `/sql` 接口。它读取 ZooKeeper 中的分片元数据和在线节点，负责 SQL 路由。
- DataNode：数据节点，对 Coordinator 暴露 HTTP `/execute`、`/wal`、`/health`。每个 DataNode 维护本地 WAL 和 MiniSQL 数据目录。
- ZooKeeper：保存集群元数据和在线节点。DataNode 使用 ephemeral znode 注册，节点宕机后会自动从在线列表消失。

MiniSQL 内核保持不变，由 Java DataNode 通过子进程方式调用。这样可以把分布式逻辑、WAL、容错恢复全部放在 `minisql/` 外部，符合“数据库内核无日志和事务系统，无需实现分布式事务”的要求。

## 3. 元数据设计

ZooKeeper 路径如下：

- `/dis-minisql/nodes/<nodeId>`：DataNode 在线信息，ephemeral 节点，内容为 `NodeInfo` JSON。
- `/dis-minisql/shards/<shardId>`：分片副本信息，persistent 节点，内容为 `ShardMetadata` JSON。

分片初始化采用确定性轮转方式。例如三节点、三分片、副本数为二时：

- shard 0: node-a, node-b
- shard 1: node-b, node-c
- shard 2: node-c, node-a

该方式保证副本均匀分布，且无需额外主节点选举即可进行简化部署。

## 4. 分片与复制

Coordinator 选择分片键后计算：

```text
shardId = sha256(shardKey) mod shardCount
```

对于 `insert into ... values (...)`，如果用户未显式传入 `shardKey`，系统默认取第一列值作为分片键；也可以在 HTTP 请求中传入 `shardKey`。

写请求流程：

1. Coordinator 根据分片键定位 shard。
2. 从 ZooKeeper 获取该 shard 的副本列表和当前在线节点。
3. 将写请求发送给所有在线副本。
4. DataNode 先追加本地 `wal.jsonl`，再调用 MiniSQL 执行。
5. 至少一个副本成功时，Coordinator 返回成功；失败副本恢复后通过 WAL 追赶。

DDL 语句如 `create table`、`drop table` 会广播到所有在线 DataNode，保证各分片本地 schema 一致。

## 5. WAL 与容错恢复

DataNode 的 WAL 为 JSON Lines 文件，每条记录包含：

- `sequence`：本节点递增序列号。
- `requestId`：Coordinator 为一次写入生成的全局请求 ID。同一次写入发往多个副本时使用相同 ID，用于恢复去重。
- `shardId`：该条日志所属分片，DDL 广播日志使用 `-1`。
- `sql`：规范化后的 SQL。
- `timestampMillis`：写入时间。

由于原 MiniSQL 启动时不会自动加载已有数据库列表，DataNode 在执行 SQL 前会重放 WAL，再执行本次 SQL。这样即使 DataNode 进程宕机重启，也能从 WAL 恢复逻辑状态。

恢复流程：

1. DataNode 启动并在 ZooKeeper 注册。
2. 根据 ZooKeeper 分片表找到与自己有共同分片的在线副本。
3. 调用对方 `/wal` 接口拉取 WAL。
4. 只保留 `shardId` 属于本节点负责分片或 `shardId = -1` 的广播日志。
5. 按 `requestId` 去重后追加本地缺失日志。
6. 后续请求通过 WAL 重放恢复为一致状态。

该设计面向“节点无正在执行任务时宕机，恢复后补执行离线期间其他副本已执行操作”的场景。它不处理单条 SQL 执行到一半时进程崩溃的事务原子性问题。系统实现最终一致性；强一致性需要多数派提交、日志复制协议或事务提交协议，考虑到原 MiniSQL 无事务系统，本实现采用更适合作业规模的 GFS/BigTable 风格简化方案。

## 6. 查询执行

读请求支持两种模式：

- 指定 `shardKey`：Coordinator 只访问对应分片的一个在线副本。
- 不指定 `shardKey`：Coordinator 对所有分片做 scatter-gather，每个分片选择一个在线副本执行，最终把多个节点的输出一起返回。

负载均衡使用 per-shard 轮询策略。每次读同一分片时，Coordinator 会在在线副本之间轮转，避免读请求集中到单个节点。

复杂查询采用中心化协调模式。当前实现将 SQL 下发到各分片并返回分片结果；对于 join 等跨分片查询，可在 Coordinator 层进一步解析结果并做内存 join。本实现保留了中心化 Coordinator 的扩展位置，满足简化 BigTable/GFS 风格实现要求。

## 7. 部署

Java 工程位于 `dis-minisql/`，示例配置为 `dis-minisql/src/main/resources/cluster.example.json`。

构建 MiniSQL：

```bash
cd minisql
mkdir -p build
cd build
cmake ..
make -j
```

构建 Java：

```bash
cd dis-minisql
mvn package
```

启动：

```bash
./scripts/start-zookeeper.sh
java -jar target/dis-minisql-1.0.0.jar init-zk cluster.json
./scripts/start-node.sh cluster.json node-a
./scripts/start-node.sh cluster.json node-b
./scripts/start-node.sh cluster.json node-c
./scripts/start-coordinator.sh cluster.json
```

三台真机部署时，只需要把 `cluster.json` 中每个节点的 `host` 改为对应机器 IP，并在每台机器启动自己的 DataNode。

## 8. 局限与改进

- 当前系统实现最终一致性，不实现分布式事务。
- WAL 恢复以 SQL 重放为核心，适合教学 MiniSQL；生产系统应使用幂等日志、LSN、快照和压缩。
- 当前复杂查询返回各分片结果，完整跨分片 join 可在 Coordinator 层继续实现结果解析与内存 join。
- 写请求在至少一个副本成功时返回成功，若需要更强一致性，可改为多数派或全部副本确认。
